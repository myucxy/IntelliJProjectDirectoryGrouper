# 分组规则与交互

## 1. 统一输入模型

所有界面先转换为与 IntelliJ UI 无关的模型：

```text
ProjectEntry
  stableId       稳定标识，优先使用规范化项目路径
  displayName    平台提供的项目显示名
  rawLocation    原始路径或非本地位置标识
  localPath      可解析时的本地 Path，否则为空
  sourceIndex    平台原始列表位置
  lastOpenedAt   最近打开时间，可为空
  payload        原平台动作或 Project 引用
```

分组算法只依赖 `localPath`、`sourceIndex` 和可选的 `lastOpenedAt`，不得扫描项目内容。

## 2. 路径规范化

处理步骤：

1. 仅处理绝对本地文件系统路径。
2. 调用 `normalize()` 消除 `.` 和 `..`，但默认不调用 `toRealPath()`，避免菜单展开时磁盘访问和符号链接语义变化。
3. 统一分隔符用于持久键；显示时使用系统分隔符。
4. Windows 默认不区分大小写；macOS 按平台默认文件系统行为不区分大小写；Linux 区分大小写。
5. UNC 路径保留服务器和共享名；文件系统根目录没有父目录时保持未分组。
6. `remote://`、Gateway、WSL/EEL 或第三方 provider 条目无法可靠映射为本地 `Path` 时，首版保持未分组。

规范化失败只影响当前条目。

## 3. 建组规则

分组键是项目根路径的直接父目录。

示例：

| 项目路径 | 分组键 | 结果 |
| --- | --- | --- |
| `D:\work\shop\api` | `D:\work\shop` | `shop` 组候选 |
| `D:\work\shop\web` | `D:\work\shop` | 与 `api` 同组 |
| `D:\work\tools\cli` | `D:\work\tools` | 若只有这一项则不建组 |
| `/home/me/acme/service-a` | `/home/me/acme` | `acme` 组候选 |
| `remote://host/project` | 无 | 未分组 |

只有同一分组键中有至少两个可见项目时才产生组。该阈值固定为 2，首版不增加设置项。

## 4. 组名

### 4.1 基础名称

使用父目录的最后一个路径片段。例如 `D:\work\shop` 显示为 `shop`。

### 4.2 同名消歧

若本次列表中两个不同父目录得到相同基础名称，追加尽可能短的上级路径：

```text
shop — work-a
shop — clients/acme
```

逐级增加上级片段，直到所有组名唯一；仍无法唯一时追加完整规范化父目录。名称计算必须确定性，相同输入得到相同名称。

### 4.3 根目录和异常名称

- 父目录是文件系统根时不建组。
- 父目录名为空时使用其系统可读路径。
- 名称不截断数据本身；UI 可以通过平台组件省略显示，并在描述或提示中提供父目录路径。

## 5. 排序

### 5.1 最近项目

- 项目原始顺序由 IntelliJ 提供，通常代表最近激活顺序。
- 一个组的排序位置等于其第一个成员在原列表中的位置。
- 组内成员保持原始相对顺序。
- 未分组条目保留各自原始位置。
- 清除列表等尾部命令保持在所有项目项之后。

### 5.2 打开项目

- 以平台当前打开项目顺序为源顺序。
- 组与成员的排序规则同上。
- 当前项目不强制置顶，避免每次切换造成菜单跳动。

### 5.3 欢迎页持久组

- 新建的插件组按其第一个成员在最近项目列表中的位置依次创建。
- 不重排已有用户组。
- 组内成员按最近项目原顺序加入。

## 6. 两类分组状态

| 场景 | 是否自动计算 | 是否持久化到 `ProjectGroup` | 是否受设置复选框控制 |
| --- | --- | --- | --- |
| 最近项目菜单 | 每次展开 | 否 | 是 |
| 打开项目菜单 | 每次展开 | 否 | 是 |
| 欢迎页原生手动组 | 用户操作 | 是 | 否 |
| 欢迎页“按父目录自动分组” | 用户点击后 | 确认后是 | 否 |

这是产品的关键隔离规则。实现时不得为了复用平台渲染而在后台把虚拟组写入 `RecentProjectsManager.groups`。

## 7. 设置界面

路径：`Settings/Preferences > Tools > Project Directory Grouper`

控件：

- 复选框：`在“最近的项目”和“打开的项目”中按父目录自动分组`

说明文字只放在设置页：

`同一直接父目录下至少有两个项目时显示为一组。此设置不会修改欢迎页中的手动项目组。`

设置默认开启。点击 Apply 后，下一次打开菜单立即使用新值。

## 8. 欢迎页自动分组流程

### 8.1 入口

- 主要入口：欢迎页 Projects 状态的快捷操作 `按父目录自动分组...`。
- 备用入口：最近项目树的上下文菜单中使用同一动作。
- 无可处理项目时动作禁用，并提供平台标准禁用说明。

### 8.2 预览对话框

标题：`按父目录自动分组`

内容至少包括：

- 将创建的组及每组项目数量。
- 将更新的插件管理组数量。
- 因已在用户组中而跳过的项目数量。
- 不产生组的单项目数量。

主按钮：`应用分组`
次按钮：`取消`

预览应是计算结果的不可编辑摘要。首版不在对话框中实现复杂的逐项勾选。

### 8.3 应用规则

1. 读取当前原生组和插件所有权状态。
2. 识别用户组：非插件创建的组，以及插件创建后被用户重命名或加入额外成员的组。
3. 从候选集中排除所有用户组成员。
4. 为剩余项目计算父目录组。
5. 对插件仍拥有的组做增量同步。
6. 对新分组键创建组；发生名称冲突时使用 `名称 (自动)`，仍冲突则追加最短路径。
7. 先移除失效成员，再删除确认已空的旧插件组；禁止直接删除包含项目的组。
8. 创建或更新后的组默认展开。

### 8.4 所有权判定

插件状态保存以下快照：

```text
ManagedGroupRecord
  parentKey
  createdName
  expectedMembers
```

当前原生组必须同时满足以下条件，插件才继续管理：

- 名称等于 `createdName`。
- 当前成员列表与上次成功应用后保存的 `expectedMembers` 完全一致，包括顺序。
- 当前成员的父目录仍等于 `parentKey`。

任一条件不满足即放弃所有权，不自动修改该组。放弃所有权只删除插件自己的记录，不删除原生组。

## 9. 状态与异常交互

| 状态 | 行为 |
| --- | --- |
| 0 个项目 | 欢迎页动作禁用 |
| 全部是单项目父目录 | 显示“没有可自动分组的项目” |
| 全部已在用户组 | 显示跳过数量，不修改状态 |
| 部分路径失效 | 仍展示原平台动作，失效处理交给原动作 |
| 平台适配器不可用 | 使用原生扁平列表，并仅显示一次兼容性通知 |
| 应用中发生并发变化 | 应用前重新读取状态；计划已过期则重新计算并要求再次确认 |
