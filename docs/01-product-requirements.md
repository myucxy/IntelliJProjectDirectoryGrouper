# 产品需求

## 1. 背景与目标

当一个父目录下包含多个可独立打开的模块项目时，IntelliJ IDE 的最近项目菜单和已打开项目菜单通常以扁平列表展示。项目名相似、数量较多或存在同名项目时，用户很难快速判断它们属于哪个工作区。

本插件通过父目录建立清晰的视觉层级，降低查找和切换项目的成本，并在欢迎页提供受用户控制的一次性自动整理能力。

成功标准：

- 用户能在两次点击内从父目录组进入目标项目。
- 安装后无需配置，IDE 内两个目标列表即按规则分组。
- 关闭自动分组后，列表恢复平台原有的扁平行为。
- 欢迎页的自动整理永远不会在未触发时修改原生项目组。
- 用户的手动项目组不会被自动整理覆盖。

## 2. 术语和界面范围

| 术语 | 本文定义 |
| --- | --- |
| 项目路径 | IDE 最近项目记录或已打开 `Project` 对应的本地项目根路径 |
| 父目录 | 项目路径的直接上一级目录，不递归寻找 Git 根目录或构建根目录 |
| 虚拟分组 | 仅在动作列表生成时计算，不写入 IntelliJ 的 `ProjectGroup` 状态 |
| 原生项目组 | IntelliJ 欢迎页使用的 `ProjectGroup` 持久化数据 |
| 最近的项目列表 | `File > Open Recent` 对应的平台动作组，动作 ID 为 `RecentProjectListGroup` |
| 打开的项目列表 | IDE 用于切换当前已打开项目窗口的平台动作组，动作 ID 为 `OpenProjectWindows` |
| 欢迎页 | 未打开项目时的 Welcome Screen / Projects 页面 |

如果某个产品使用不同的本地化文案，仍按动作 ID 和行为确定目标界面，不按显示文本匹配。

## 3. 用户故事

### US-01 最近项目自动分组

作为同时维护多个仓库和模块的开发者，我希望最近项目按直接父目录展示，以便先找到工作区，再找到项目。

### US-02 打开项目自动分组

作为同时打开多个项目窗口的开发者，我希望项目切换列表按父目录展示，以便快速切到同一工作区中的另一个模块。

### US-03 可关闭

作为习惯平台原始列表的用户，我希望在设置中关闭自动分组，并立即恢复原有列表行为。

### US-04 欢迎页手动控制

作为重视自定义结构的用户，我希望欢迎页保留新建组、移动到组、编辑组等原生手动能力，插件不在后台修改它。

### US-05 欢迎页一键整理

作为首次使用项目组的用户，我希望在欢迎页手动触发按父目录自动分组，预览结果并确认后完成整理。

## 4. 功能需求

### FR-01 设置项

- 在 `Settings/Preferences > Tools > Project Directory Grouper` 提供复选框：`在“最近的项目”和“打开的项目”中按父目录自动分组`。
- 初始默认值为选中。
- 设置为应用级，对同一 IDE 配置中的所有项目生效。
- 修改后不要求重启 IDE；下一次展开目标菜单时生效。
- 该设置只控制两个 IDE 内列表，不控制欢迎页按钮是否可用。

### FR-02 最近项目虚拟分组

- 设置开启时，读取平台原本要显示的最近项目动作及其项目路径。
- 使用 [02-grouping-and-ux.md](02-grouping-and-ux.md) 中的统一规则生成父目录组。
- 组内项目动作必须保留平台原动作的打开、移除失效项目、图标、描述和快捷键行为。
- 平台提供的非本地或第三方最近项目动作无法获得本地路径时，保留为未分组项。
- “清除最近项目列表”等非项目动作必须保留原位置和行为。
- 设置关闭时，返回平台原动作组的内容与顺序，不产生分组。

### FR-03 已打开项目虚拟分组

- 设置开启时，对当前已打开且具有本地路径的项目按同一规则分组。
- 点击项目项后应激活对应项目窗口；最小化窗口应按平台能力恢复并获得焦点。
- 当前项目应保留平台可识别的选中状态。
- 无本地路径的轻量编辑、远程前端或特殊项目作为未分组项保留。
- 项目打开、关闭或重命名后，下次展开列表时必须反映最新状态。
- 设置关闭时，委托给原生动作组，不改变平台行为。

### FR-04 欢迎页保留手动分组

- 不替换 IntelliJ 原生的“新建组”“移动到组”“编辑组”等操作。
- 插件后台刷新、IDE 启动和最近项目变化时，不得写入原生项目组。
- 用户手动组的名称、成员、顺序和展开状态应保持不变。

### FR-05 欢迎页手动自动分组

- 在欢迎页有项目时，提供 `按父目录自动分组...` 操作。
- 操作至少出现在欢迎页的项目相关快捷操作区；同时加入最近项目的上下文操作组作为备用入口。
- 点击后先计算计划，再显示确认对话框，包含：将创建/更新的组数、将移动的项目数、因已有手动组而跳过的项目数。
- 确认后才调用 IntelliJ 原生项目组 API。
- 只移动当前未处于用户手动组中的项目；插件上次创建且仍由插件管理的组可以增量更新。
- 重复执行必须幂等：输入未变化时，不新增重复组，不改变用户组。
- 执行完成后显示简短结果通知，并刷新欢迎页。
- 没有任何父目录包含两个及以上可处理项目时，操作禁用或显示“没有可自动分组的项目”，不得创建空组。

### FR-06 插件生成组的所有权

- 插件使用应用级持久状态记录自己创建的组标识、当时使用的名称、父目录键和成员快照。
- 用户重命名插件生成的组后，该组立即视为用户维护；后续自动执行不得重命名或删除它。
- 用户手动增加、移除或重排插件生成组的成员后，该组视为用户维护。
- 清理插件生成组时，必须先移出成员，再删除空组，避免平台删除组时影响最近项目元数据。
- 卸载插件不删除任何原生项目组；用户仍可继续使用这些组。

### FR-07 错误与降级

- 某个平台版本不支持安全接管目标动作组时，插件必须保留原生列表，而不是让菜单消失或报错。
- 单条无效路径不能阻止其他项目显示。
- 路径解析失败应记录调试日志；界面只显示一次简洁通知，不连续弹窗。
- 欢迎页写入部分失败时，显示完成数与失败数，不声称全部成功。

## 5. 非功能需求

### NFR-01 兼容性

- 只依赖 IntelliJ Platform 模块，不依赖 Java、Python、Android 或其他语言插件。
- 目标产品包括 IntelliJ IDEA、Android Studio、PyCharm、WebStorm、PhpStorm、RubyMine、GoLand、CLion、DataGrip 和 Rider；以 Plugin Verifier 与人工验证结果决定 Marketplace 实际兼容声明。
- 首版建议覆盖 2024.2 至 2026.2，对应 build 242、251、252、261、262；262 的精确验收构建为 `IU-262.8665.258`。若平台内部 API 不兼容，应使用版本适配源集或拆分发布物。

### NFR-02 性能

- 菜单展开时分组计算不访问网络、不递归扫描目录、不读取项目文件。
- 1000 条最近项目记录的纯分组计算目标为 50 ms 以内；UI 动作构建目标为 150 ms 以内。
- Action `update()` 不执行磁盘 I/O，遵守平台 BGT/EDT 线程要求。

### NFR-03 隐私与安全

- 不发送项目名、路径、组名或使用数据到网络。
- 插件状态仅保存设置和插件生成组的必要元数据。
- 日志默认不输出完整用户路径；调试日志可使用脱敏后的路径哈希或仅输出异常类型。

### NFR-04 可卸载性

- 支持动态卸载；卸载时恢复被替换的平台动作。
- 卸载不要求重启，不留下动作 ID 冲突或不可用菜单。

### NFR-05 可访问性与本地化

- 所有文案从 resource bundle 读取，首版至少提供英文与简体中文。
- 不只依靠颜色表示组、当前项目或错误状态。
- 操作可通过键盘访问，组名称和项目描述应被辅助技术读取。

## 6. 不在首版范围

- 自定义分组层级、正则规则、Git 仓库根目录识别或 Maven/Gradle 模块识别。
- 多级嵌套父目录组。
- 自动扫描某个磁盘目录并导入项目。
- 云端同步插件私有配置。
- 修改 JetBrains 原生组的拖拽行为。
- 给项目路径打标签、颜色或收藏标记。

## 7. 发布级验收标准

- AC-01：全新安装后设置默认开启，两个目标列表中同父目录的两个项目显示在同一组下。
- AC-02：只有一个项目的父目录不生成组。
- AC-03：关闭设置后两个列表恢复平台原始顺序和行为，欢迎页组不发生变化。
- AC-04：开启设置不会自动新增、删除或移动欢迎页原生项目组。
- AC-05：欢迎页自动分组在确认前不写入状态，取消后状态完全不变。
- AC-06：欢迎页自动分组跳过用户手动组成员，重复运行不产生重复组。
- AC-07：同名父目录使用可区分且稳定的显示名称。
- AC-08：Windows、macOS、Linux 的路径比较符合各自默认文件系统大小写规则。
- AC-09：无本地路径或第三方最近项目项仍然可见且可执行。
- AC-10：插件动态卸载后原生菜单仍可用。
- AC-11：目标 build 与产品矩阵通过 Plugin Verifier，无兼容性错误。
- AC-12：1000 条数据的纯算法性能满足 NFR-02。
